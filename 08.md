# 介绍情境检索（Contextual Retrieval）

发表于 2024 年 9 月 19 日

对于 AI 模型在特定场景中发挥作用，它通常需要访问背景知识。

对于 AI 模型在特定场景中发挥作用，它通常需要访问背景知识。例如，客户支持聊天机器人需要了解它们所服务的具体业务，而法律分析机器人需要知道大量过去的案例。

开发者通常使用检索增强生成（RAG）来增强 AI 模型的知识。RAG 是一种从知识库中检索相关信息并将其附加到用户提示中的方法，显著增强了模型的响应。问题是，传统的 RAG 解决方案在编码信息时会移除上下文，这通常导致系统无法从知识库中检索到相关信息。

在本文中，我们介绍了一种可以显著改善 RAG 中检索步骤的方法。该方法称为"情境检索（Contextual Retrieval）"，使用两种子技术：情境嵌入（Contextual Embeddings）和情境 BM25（Contextual BM25）。该方法可以将检索失败次数减少 49%，与重新排序（reranking）结合使用时可减少 67%。这代表了检索准确性的显著改进，直接转化为下游任务的更好性能。

你可以使用 [我们的示例集](https://platform.claude.com/cookbook/capabilities-contextual-embeddings-guide) 轻松部署自己的情境检索解决方案。

### 关于简单使用更长提示的说明

有时最简单的解决方案就是最好的。如果你的知识库小于 200,000 个 token（大约 500 页材料），你可以将整个知识库包含在提供给模型的提示中，无需 RAG 或类似方法。

几周前，我们为 Claude 发布了 [提示缓存（prompt caching）](https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching)，这使得这种方法显著更快且更具成本效益。开发者现在可以在 API 调用之间缓存常用提示，将延迟降低 2 倍以上，成本降低高达 90%（通过阅读我们的 [提示缓存示例集](https://platform.claude.com/cookbook/misc-prompt-caching) 了解其工作原理）。

然而，随着知识库的增长，你需要一个更具可扩展性的解决方案。这就是情境检索发挥作用的地方。

## RAG 入门：扩展到更大的知识库

对于不适合上下文窗口的更大知识库，RAG 是典型解决方案。RAG 通过使用以下步骤预处理知识库来工作：

1. 将知识库（文档的"语料库"）分解为较小的文本块，通常不超过几百个 token；
2. 使用嵌入模型将这些块转换为编码含义的向量嵌入；
3. 将这些嵌入存储在允许按语义相似性搜索的向量数据库中。

在运行时，当用户向模型输入查询时，向量数据库用于根据与查询的语义相似性找到最相关的块。然后，最相关的块被添加到发送给生成模型的提示中。

虽然嵌入模型擅长捕捉语义关系，但它们可能会错过关键的精确匹配。幸运的是，有一种较老的技术可以在这些情况下提供帮助。BM25（Best Matching 25）是一种使用词法匹配来查找精确单词或短语匹配的排序函数。它对于包含唯一标识符或技术术语的查询特别有效。

BM25 通过建立在 TF-IDF（词频 - 逆文档频率）概念上工作。TF-IDF 测量一个词在集合中对文档的重要性。BM25 通过考虑文档长度并对词频应用饱和函数来改进这一点，这有助于防止常见词主导结果。

以下是 BM25 如何在语义嵌入失败的地方成功：假设用户在技术支持数据库中查询"错误代码 TS-999"。嵌入模型可能会找到关于错误代码的一般内容，但可能会错过精确的"TS-999"匹配。BM25 查找此特定文本字符串以识别相关文档。

RAG 解决方案可以通过结合嵌入和 BM25 技术使用以下步骤更准确地检索最适用的块：

1. 将知识库（文档的"语料库"）分解为较小的文本块，通常不超过几百个 token；
2. 为这些块创建 TF-IDF 编码和语义嵌入；
3. 使用 BM25 基于精确匹配查找顶级块；
4. 使用嵌入基于语义相似性查找顶级块；
5. 使用排名融合技术组合并去重（3）和（4）的结果；
6. 将 top-K 块添加到提示中生成响应。

通过利用 BM25 和嵌入模型，传统 RAG 系统可以提供更全面和准确的结果，平衡精确术语匹配与更广泛的语义理解。

这种方法允许你以具有成本效益的方式扩展到巨大的知识库，远远超出单个提示能容纳的范围。但这些传统 RAG 系统有一个重大限制：它们经常破坏上下文。

### 传统 RAG 中的上下文困境

在传统 RAG 中，文档通常被分割成较小的块以进行有效检索。虽然这种方法对许多应用有效，但当单个块缺乏足够上下文时，它可能导致问题。

例如，想象你的知识库中嵌入了财务信息集合（比如美国 SEC 文件），你收到以下问题：*"ACME 公司 2023 年第二季度的收入增长是多少？"*

相关块可能包含文本：*"该公司收入比上一季度增长 3%。"* 然而，这个块本身没有指定指的是哪家公司或相关时间段，使得难以检索正确信息或有效使用信息。

## 介绍情境检索

情境检索通过在嵌入前（"情境嵌入"）和创建 BM25 索引前（"情境 BM25"）在每个块前添加块特定解释性上下文来解决这个问题。

让我们回到 SEC 文件集合的例子。以下是块如何转换的示例：

```text
original_chunk = "该公司收入比上一季度增长 3%。"

contextualized_chunk = "这个块来自 ACME 公司 2023 年第二季度业绩的 SEC 文件；上一季度收入为 3.14 亿美元。该公司收入比上一季度增长 3%。"
```

值得注意的是，过去已经提出了其他使用上下文改善检索的方法。其他提议包括：[向块添加通用文档摘要](https://aclanthology.org/W02-0405.pdf)（我们实验发现增益非常有限），[假设文档嵌入](https://arxiv.org/abs/2212.10496)，和 [基于摘要的索引](https://www.llamaindex.ai/blog/a-new-document-summary-index-for-llm-powered-qa-systems-9a32ece2f9ec)（我们评估发现性能较低）。这些方法不同于本文提出的方法。

### 实现情境检索

当然，手动标注知识库中数千甚至数百万个块的工作量太大了。为了实现情境检索，我们转向 Claude。我们编写了一个提示，指示模型提供简洁的块特定上下文，使用整个文档的上下文来解释该块。我们使用以下 Claude 3 Haiku 提示为每个块生成上下文：

```text
<document> 
{{WHOLE_DOCUMENT}} 
</document> 
这里是我们想要置于整个文档中的块
<chunk> 
{{CHUNK_CONTENT}} 
</chunk> 
请给出一个简短简洁的上下文，将这个块置于整个文档中，以改善块的搜索检索。只回答简洁的上下文，不要其他内容。 
```

生成的上下文文本（通常 50-100 个 token）在嵌入和创建 BM25 索引前附加到块前。

如果你对使用情境检索感兴趣，可以从 [我们的示例集](https://platform.claude.com/cookbook/capabilities-contextual-embeddings-guide) 开始。

### 使用提示缓存降低情境检索成本

情境检索在 Claude 上可以以低成本实现，这要归功于上面提到的特殊提示缓存功能。使用提示缓存，你不需要为每个块传入参考文档。你只需将文档加载到缓存一次，然后引用之前缓存的内容。假设 800 token 块、8k token 文档、50 token 上下文指令和每个块 100 token 上下文，**生成情境化块的一次性成本为每百万文档 token 1.02 美元**。

#### 方法

我们在各种知识领域（代码库、小说、ArXiv 论文、科学论文）、嵌入模型、检索策略和评估指标进行了实验。我们在 [附录 II](https://assets.anthropic.com/m/1632cded0a125333/original/Contextual-Retrieval-Appendix-2.pdf) 中包含了每个领域使用的问答示例。

下图显示了在所有知识领域中具有最佳性能的嵌入配置（Gemini Text 004）并检索 top-20-chunks 的平均性能。我们使用 1 减去 recall@20 作为评估指标，测量相关文档在前 20 个块中未能被检索的百分比。你可以在附录中看到完整结果——情境化在我们评估的每个嵌入源组合中都提高了性能。

#### 性能改进

我们的实验显示：

- **情境嵌入将 top-20-chunk 检索失败率降低了 35%**（5.7% → 3.7%）。
- **结合情境嵌入和情境 BM25 将 top-20-chunk 检索失败率降低了 49%**（5.7% → 2.9%）。

#### 实现考虑

在实现情境检索时，需要注意以下几点：

1. **块边界**：考虑如何将文档分割成块。块大小、块边界和块重叠的选择可能影响检索性能。
2. **嵌入模型**：虽然情境检索提高了我们测试的所有嵌入模型的性能，但有些模型可能比其他模型受益更多。我们发现 [Gemini](https://ai.google.dev/gemini-api/docs/embeddings) 和 [Voyage](https://www.voyageai.com/) 嵌入特别有效。
3. **自定义情境化提示**：虽然我们提供的通用提示效果很好，但你可以通过针对特定领域或使用案例定制的提示获得更好的结果（例如，包括可能仅在知识库其他文档中定义的关键术语词汇表）。
4. **块数量**：向上下文窗口添加更多块增加了包含相关信息的机会。然而，更多信息可能会分散模型的注意力，所以这有一个限度。我们尝试提供 5、10 和 20 个块，发现使用 20 个是这些选项中性能最好的（参见附录比较），但在你的使用案例上值得实验。

**始终运行评估**：通过传递情境化块并区分什么是上下文和什么是块，可以改善响应生成。

## 通过重新排序进一步提升性能

在最后一步中，我们可以将情境检索与另一种技术结合以获得更多性能改进。在传统 RAG 中，AI 系统搜索其知识库以找到潜在相关信息块。对于大型知识库，此初始检索通常返回大量块——有时数百个——具有不同的相关性和重要性。

重新排序是一种常用的过滤技术，确保只有最相关的块传递给模型。重新排序提供更好的响应并降低成本和延迟，因为模型处理的信息更少。关键步骤是：

1. 执行初始检索以获得潜在相关块（我们使用前 150）；
2. 将 top-N 块和用户的查询传递给重新排序模型；
3. 使用重新排序模型，根据每个块与提示的相关性和重要性打分，然后选择 top-K 块（我们使用前 20）；
4. 将 top-K 块作为上下文传递给模型生成最终结果。

### 性能改进

市场上有几个重新排序模型。我们使用 [Cohere 重新排序器](https://cohere.com/rerank) 进行测试。Voyage [也提供重新排序器](https://docs.voyageai.com/docs/reranker)，尽管我们没有时间测试它。我们的实验显示，在各个领域中，添加重新排序步骤进一步优化了检索。

具体来说，我们发现重新排序的情境嵌入和情境 BM25 将 top-20-chunk 检索失败率降低了 67%（5.7% → 1.9%）。

#### 成本和延迟考虑

重新排序的一个重要考虑因素是对延迟和成本的影响，特别是在重新排序大量块时。因为重新排序在运行时添加了额外步骤，它不可避免地添加了一点延迟，即使重新排序器并行对所有块评分。在为了更好性能重新排序更多块与为了更低延迟和成本重新排序更少块之间存在固有权衡。我们建议在你的特定使用案例上实验不同设置以找到正确的平衡。

## 结论

我们运行了大量测试，比较了上述所有技术的不同组合（嵌入模型、BM25 使用、情境检索使用、重新排序器使用以及检索的 top-K 结果总数），跨越各种数据集类型。以下是我们发现的总结：

1. Embeddings+BM25 优于单独的 embeddings；
2. Voyage 和 Gemini 在我们测试的模型中具有最好的嵌入；
3. 向模型传递 top-20 块比仅传递 top-10 或 top-5 更有效；
4. 向块添加上下文大大改善了检索准确性；
5. 重新排序优于不重新排序；
6. **所有这些好处可以叠加**：为了最大化性能改进，我们可以将情境嵌入（来自 Voyage 或 Gemini）与情境 BM25 结合，加上重新排序步骤，并将 20 个块添加到提示中。

我们鼓励所有使用知识库的开发者使用 [我们的示例集](https://platform.claude.com/cookbook/capabilities-contextual-embeddings-guide) 实验这些方法以解锁新的性能水平。
