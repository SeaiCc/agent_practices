> 扩展思考功能自初始发布以来已有改进，因此在大多数情况下我们建议使用该功能而不是专用的思考工具。扩展思考提供类似的好处——给 Claude 推理复杂问题的空间——同时具有更好的集成和性能。详见我们的扩展思考文档了解实现细节。

随着我们继续增强 Claude 的复杂问题解决能力，我们发现了一种特别有效的方法："思考"工具，它在复杂任务期间为结构化思考创建专用空间。

这种简单而强大的技术——如下文所述，它与 Claude 的新"扩展思考"能力不同（见扩展思考实现细节）——已在 Claude 的智能体工具使用能力方面取得了显著改进。这包括遵循策略、做出一致决策和处理多步骤问题，所有这些都只需最小的实现开销。

在本文中，我们将探讨如何在不同应用中实现"思考"工具，基于经验证的基准测试结果与开发者分享实用指导。

### 什么是"思考"工具？

使用"思考"工具，我们让 Claude 能够在获得最终答案的过程中包含额外的思考步骤——拥有自己指定的空间。

虽然它听起来与扩展思考相似，但它是不同的概念。扩展思考关乎 Claude 在开始生成响应之前做什么。使用扩展思考，Claude 在采取行动之前深入考虑并迭代其计划。"思考"工具是用于 Claude 一旦开始生成响应，添加一个步骤停下来思考它是否拥有继续前进所需的所有信息。这在执行长工具调用链或与用户的长多步骤对话时特别有用。

这使得"思考"工具更适合 Claude 没有从用户查询本身制定响应所需的所有信息的场景，以及需要处理外部信息（例如工具调用结果中的信息）的场景。Claude 使用"思考"工具执行的推理不如扩展思考所能获得的那么全面，更专注于模型发现的*新*信息。

我们建议对更简单的工具使用场景使用扩展思考，如非顺序工具调用或直接的指令遵循。扩展思考也适用于你不需要 Claude 调用工具的使用场景，如编码、数学和物理。"思考"工具更适合 Claude 需要调用复杂工具、在长工具调用链中仔细分析工具输出、在具有详细指南的政策密集型环境中导航，或做出顺序决策，其中每个步骤都建立在前一个步骤之上且错误代价高昂。

以下是使用来自τ-Bench 的标准工具规范格式的示例实现：

```json
{
  "name": "think",
  "description": "使用工具思考某事。它不会获取新信息或更改数据库，只是将思考追加到日志。在需要复杂推理或一些缓存内存时使用它。",
  "input_schema": {
    "type": "object",
    "properties": {
      "thought": {
        "type": "string",
        "description": "要思考的想法。"
      }
    },
    "required": ["thought"]
  }
}
```

### 在τ-Bench 上的性能

我们使用τ-bench（tau-bench）评估了"思考"工具，这是一个全面的基准测试，旨在测试模型在现实客户服务场景中使用工具的能力，"思考"工具是评估标准环境的一部分。

τ-bench 评估 Claude 的能力：

- 与模拟用户导航真实对话
- 一致遵循复杂的客户服务代理策略指南
- 使用各种工具访问和操作环境数据库

τ-bench 中使用的主要评估指标是 pass^*k*，它衡量给定任务的所有*k*个独立任务试验成功的概率，在所有任务上平均。与其他 LLM 评估中常见的 pass@*k* 指标（衡量*k*个试验中至少一个成功）不同，pass^*k*评估一致性和可靠性——这是客户服务应用的关键品质，一致遵循策略至关重要。

#### 性能分析

我们的评估比较了几种不同的配置：

1. 基线（无"思考"工具，无扩展思考模式）
2. 仅扩展思考模式
3. 仅"思考"工具
4. "思考"工具配合优化提示（针对航空域）

结果显示，当 Claude 3.7 在基准测试的"航空"和"零售"客户服务域中有效使用"思考"工具时，性能大幅提升：

- **航空域**：配合优化提示的"思考"工具在 pass^1 指标上达到 0.570，而基线仅为 0.370——相对提升 54%；
- **零售域**：仅"思考"工具达到 0.812，而基线为 0.783。

Claude 3.7 Sonnet 在 Tau-Bench 评估"航空"域的性能

| 配置 | *k*=1 | *k*=2 | *k*=3 | *k*=4 | *k*=5 |
|------|-------|-------|-------|-------|-------|
| "思考" + 提示 | 0.584 | 0.444 | 0.384 | 0.356 | 0.340 |
| "思考" | 0.404 | 0.254 | 0.186 | 0.140 | 0.100 |
| 扩展思考 | 0.412 | 0.290 | 0.232 | 0.192 | 0.160 |
| 基线 | 0.332 | 0.206 | 0.148 | 0.116 | 0.100 |

航空域的最佳性能是通过将"思考"工具与优化提示配对实现的，该提示给出分析客户请求时使用的推理方法类型的示例。以下是优化提示的示例：

```markdown
## 使用思考工具

在采取任何行动或在接收工具结果后响应用户之前，使用思考工具作为草稿纸：
- 列出适用于当前请求的特定规则
- 检查是否已收集所有必需信息
- 验证计划行动符合所有策略
- 迭代工具结果以检查正确性

以下是思考工具内迭代内容的示例：
<think_tool_example_1>
用户想取消航班 ABC123
- 需要验证：用户 ID、预订 ID、原因
- 检查取消规则：
  * 是否在预订 24 小时内？
  * 如果不是，检查机票类别和保险
- 验证无段已飞行或在过去
- 计划：收集缺失信息，验证规则，获取确认
</think_tool_example_1>

<think_tool_example_2>
用户想预订 3 张去 NYC 的机票，每人 2 件托运行李
- 需要用户 ID 以检查：
  * 行李额度的会员等级
  * 个人资料中存在哪些支付方式
- 行李计算：
  * 经济舱 × 3 位乘客
  * 如果是普通会员：每人 1 件免费行李 → 3 件额外行李 = $150
  * 如果是银卡会员：每人 2 件免费行李 → 0 件额外行李 = $0
  * 如果是金卡会员：每人 3 件免费行李 → 0 件额外行李 = $0
- 支付规则需要验证：
  * 最多 1 张旅行凭证、1 张信用卡、3 张礼品卡
  * 所有支付方式必须在个人资料中
  * 旅行凭证余额作废
- 计划：
1. 获取用户 ID
2. 验证行李费的会员等级
3. 检查个人资料中的支付方式及其组合是否允许
4. 计算总计：票价 + 任何行李费
5. 获取预订的明确确认
</think_tool_example_2>
```

特别有趣的是不同方法的比较。使用配合优化提示的"思考"工具取得了显著优于扩展思考模式的结果（扩展思考模式显示与无提示"思考"工具相似的性能）。仅使用"思考"工具（无提示）提高了基线性能，但仍低于优化方法。

"思考"工具与优化提示的组合以显著优势提供了最强的性能，这可能是由于基准测试中航空策略部分的高复杂性，模型从获得如何"思考"的示例中受益最多。

在零售域，我们也测试了各种配置以了解每种方法的具体影响。

Claude 3.7 Sonnet 在 Tau-Bench 评估"零售"域的性能

| 配置 | *k*=1 | *k*=2 | *k*=3 | *k*=4 | *k*=5 |
|------|-------|-------|-------|-------|-------|
| "思考" + 无提示 | 0.812 | 0.735 | 0.685 | 0.650 | 0.626 |
| 扩展思考 | 0.770 | 0.681 | 0.623 | 0.581 | 0.548 |
| 基线 | 0.783 | 0.695 | 0.643 | 0.607 | 0.583 |

"思考"工具即使没有额外提示也达到了最高的 pass^1 分数 0.812。零售策略明显比航空域更容易导航，Claude 仅通过拥有思考空间就能改进，无需进一步指导。

#### 来自τ-Bench 分析的关键洞察

我们的详细分析揭示了几个可以帮助有效实现"思考"工具的模式：

1. **提示在困难域中非常重要**。仅仅使"思考"工具可用可能会在一定程度上提高性能，但与优化提示配对为困难域带来了显著更好的结果。然而，较简单的域可能仅从拥有"思考"访问中受益。
2. **跨试验改进一致性**。使用"思考"的改进在 pass^k 高达 k=5 时保持，表明该工具帮助 Claude 更有效地处理边缘情况和异常场景。

### 在 SWE-Bench 上的性能

在评估 Claude 3.7 Sonnet 时，类似的"思考"工具被添加到我们的 SWE-bench 设置中，有助于实现 0.623 的最先进分数。适配的"思考"工具定义如下：

```json
{
  "name": "think",
  "description": "使用工具思考某事。它不会获取新信息或对存储库进行任何更改，只是记录思考。在需要复杂推理或头脑风暴时使用它。例如，如果你探索存储库并发现 bug 的来源，调用此工具头脑风暴几种修复 bug 的独特方法，并评估哪些更改可能最简单和最有效。或者，如果你收到一些测试结果，调用此工具头脑风暴修复失败测试的方法。",
  "input_schema": {
    "type": "object",
    "properties": {
      "thought": {
        "type": "string",
        "description": "你的思考。"
      }
    },
    "required": ["thought"]
  }
}
```

我们的实验（*n*=30 个有"思考"工具的样本，*n*=144 个无）显示包含此工具的隔离效果平均提高性能 1.6%（Welch *t* 检验：*t*(38.89) = 6.71，*p* < .001，*d* = 1.47）。

### 何时使用"思考"工具

基于这些评估结果，我们确定了 Claude 从"思考"工具中受益最多的特定场景：

1. **工具输出分析**。当 Claude 需要在行动前仔细处理先前工具调用的输出，并且可能需要回溯其方法时；
2. **政策密集型环境**。当 Claude 需要遵循详细指南并验证合规性时；以及
3. **顺序决策**。当每个行动建立在前一个行动之上且错误代价高昂时（常见于多步骤域）。

## 实现最佳实践

为了从 Claude 的"思考"工具中获得最大收益，我们建议基于τ-bench 实验的以下实现实践。

#### 1. 配合域特定示例的策略性提示

最有效的方法是提供何时以及如何使**用"思考"工具的清晰指示，如τ-bench 航空域所使用的。提供量身定制于你特定使用场景的示例显著提高模型有效使用"思考"工具的方式：

- 推理过程中期望的详细程度；
- 如何将复杂指示分解为可操作的步骤；
- 处理常见场景的决策树；以及
- 如何检查是否已收集所有必要信息。

#### 2. 将复杂指南放在系统提示中

我们发现，当它们长和/或复杂时，在系统提示中包含关于"思考"工具的指示比将它们放在工具描述本身中更有效。这种方法提供更广泛的上下文，并帮助模型更好地将思考过程集成到其整体行为中。

### 何时*不*使用"思考"工具

虽然"思考"工具可以提供实质性改进，但它不适用于所有工具使用场景，并且确实以增加的提示长度和输出令牌为代价。具体而言，我们发现"思考"工具在以下使用场景中不提供任何改进：

1. **非顺序工具调用**。如果 Claude 只需要进行单个工具调用或多个并行调用来完成任务，添加"思考"不太可能有改进。
2. **简单指令遵循**。当没有太多 Claude 需要遵循的约束，且其默认行为已经足够好时，不太可能从额外"思考"中获得收益。

### 开始使用

"思考"工具是对你的 Claude 实现的简单添加，只需几个步骤即可产生有意义的改进：

1. **使用智能体工具使用场景测试**。从挑战性使用场景开始——Claude 目前在策略合规性或长工具调用链中的复杂推理方面挣扎的场景。
2. **添加工具定义**。实现定制于你域的"思考"工具。它需要最少代码但启用更结构化的推理。还考虑在系统提示中包含何时以及如何使用该工具的指示，以及与你域相关的示例。
3. **监控和细化**。观察 Claude 在实践中如何使用该工具，并调整提示以鼓励更有效的思考模式。

最好的部分是添加此工具在性能结果方面几乎没有缺点。除非 Claude 决定使用它，否则它不会改变外部行为，也不会干扰你现有的工具或工作流。

### 结论

我们的研究证明，"思考"工具可以显著增强 Claude 3.7 Sonnet 在需要策略遵循和长工具调用链中推理的复杂任务上的性能。"思考"不是一刀切的解决方案，但它为正确的使用场景提供了实质性好处，所有这些都只需最小的实现复杂性。

我们期待看到你将如何使用"思考"工具与 Claude 构建更有能力、更可靠和更透明的 AI 系统。
