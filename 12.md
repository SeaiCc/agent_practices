# 揭秘 AI 智能体的评估方法

发表于 2026 年 1 月 9 日

使智能体有用的能力也使它们难以评估。跨部署有效的策略结合了多种技术，以匹配其所测量的系统的复杂性。

---

## 引言

良好的评估帮助团队更有信心地发布 AI 智能体。没有它们，团队很容易陷入被动循环——只在生产环境中发现问题，修复一个故障又引发其他问题。评估在问题影响用户之前使其可见，其价值在智能体生命周期中不断累积。

正如我们在《构建高效智能体》中所描述的，智能体在多轮交互中运行：调用工具、修改状态，并根据中间结果进行调整。这些使 AI 智能体有用的相同能力——自主性、智能性和灵活性——也使它们更难评估。

通过内部工作以及与智能体开发前沿的客户合作，我们学会了如何为智能体设计更严谨、更有用的评估。以下是我们在真实世界部署中，针对各种智能体架构和用例行之有效的方法。

## 评估的结构

评估（"eval"）是对 AI 系统的测试：给 AI 一个输入，然后对其输出应用评分逻辑以衡量成功。在本文中，我们关注可以在开发期间无需真实用户即可运行的自动评估。

单轮评估很简单：一个提示、一个响应和评分逻辑。对于早期的 LLM，单轮、非智能体的评估是主要评估方法。随着 AI 能力的进步，多轮评估变得越来越普遍。

在简单的评估中，智能体处理一个提示，评分器检查输出是否符合预期。对于更复杂的多轮评估，一个代码智能体接收工具、任务（在本例中是构建一个 MCP 服务器）和环境，执行"智能体循环"（工具调用和推理），并用实现更新环境。评分随后使用单元测试来验证工作的 MCP 服务器。

智能体评估甚至更复杂。智能体在多轮中使用工具，修改环境中的状态并随过程调整——这意味着错误可以传播和累积。前沿模型还可以找到超越静态评估限制的创造性解决方案。例如，Opus 4.1.3 通过发现策略中的漏洞，解决了一个关于预订航班的τ2-bench 问题。它"未能通过"如所写的评估，但实际上为用户想出了一个更好的解决方案。

在构建智能体评估时，我们使用以下定义：

- **任务**（又名问题或测试用例）是具有定义输入和成功标准的单个测试。
- **对任务的每次尝试是一次试验。**由于模型输出在运行之间有所变化，我们运行多次试验以产生更一致的结果。
- **评分器**是对智能体表现的某些方面进行评分的逻辑。一个任务可以有多个评分器，每个包含多个断言（有时称为检查）。
- **转录记录**（也称为轨迹或轨迹路径）是试验的完整记录，包括输出、工具调用、推理、中间结果和任何其他交互。对于 Anthropic API，这是评估运行结束时的完整消息数组——包含对 API 的所有调用和评估期间返回的所有响应。
- **结果**是试验结束时环境中的最终状态。一个航班预订智能体可能在转录记录结束时说"您的航班已预订"，但结果是环境中 SQL 数据库中是否存在预订。
- **评估框架**是端到端运行评估的基础设施。它提供指令和工具，并行运行任务，记录所有步骤，对输出评分，并聚合结果。
- **智能体框架**（或支架）是使模型能够作为智能体运行的系统：它处理输入，编排工具调用，并返回结果。当我们评估"一个智能体"时，我们评估的是框架和模型一起工作。例如，Claude Code 是一个灵活的智能体框架，我们通过 Agent SDK 使用其核心原语来构建长运行智能体框架。
- **评估套件**是设计用于测量特定能力或行为的任务集合。套件中的任务通常共享一个广泛的目标。例如，客户支持评估套件可能测试退款、取消和升级。

## 为什么要构建评估？

当团队开始构建智能体时，他们可以通过手动测试、自用和直觉的组合取得惊人的进展。更严格的评估甚至可能似乎像是减缓发布的额外负担。但在早期原型设计阶段之后，一旦智能体投入生产并开始扩展，没有评估的构建就开始崩溃。

崩溃点通常发生在用户报告智能体在更改后感觉变差，而团队"盲目飞行"，除了猜测和检查外没有验证方法。没有评估，调试是被动的：等待投诉，手动重现，修复错误，并希望没有其他东西退化。团队无法区分真实退化和噪声，无法在发布前自动测试数百个场景的更改，也无法衡量改进。

我们多次看到这种进程展开。例如，Claude Code 最初基于来自 Anthropic 员工和外部用户的反馈进行快速迭代。后来，我们添加了评估——首先是针对简洁性和文件编辑等狭窄领域，然后是针对过度工程等更复杂的行为。这些评估帮助识别问题、指导改进，并聚焦研究 - 产品合作。与生产监控、A/B 测试、用户研究等相结合，评估提供了继续改进 Claude Code 的信号，随着其扩展。

在智能体生命周期的任何阶段编写评估都是有用的。早期，评估迫使产品团队明确智能体的成功意味着什么，而后期它们帮助维持一致的质量标准。

Descript 的智能体帮助用户编辑视频，因此他们围绕成功编辑工作的三个维度构建评估：不要破坏东西，做我要求的事，并做好。他们从手动评分演变为由产品团队定义标准并进行定期人工校准的 LLM 评分器，现在定期运行两个单独的套件进行质量基准测试和回归测试。

Bolt AI 团队开始构建评估较晚，在他们已经拥有一个广泛使用的智能体之后。在 3 个月内，他们构建了一个评估系统，运行他们的智能体并用静态分析对输出评分，使用浏览器智能体测试应用程序，并采用 LLM 裁判进行指令遵循等行为。

一些团队在开发开始时创建评估；其他团队在大规模时添加它们，当评估成为改进智能体的瓶颈时。评估在智能体开发开始时特别有用，以明确编码期望行为。两名工程师阅读相同的初始规范可能会对 AI 如何处理边缘情况得出不同的解释。评估套件解决了这种歧义。无论何时创建，评估都有助于加速开发。

评估还塑造你采用新模型的速度。当更强大的模型出现时，没有评估的团队面临数周的测试，而有评估的竞争对手可以在几天内确定模型的优势、调整提示并升级。

一旦评估存在，你就免费获得基准和回归测试：延迟、令牌使用、每任务成本和错误率可以在静态任务库上跟踪。评估还可以成为产品和研究团队之间带宽最高的沟通渠道，定义研究人员可以优化的指标。显然，评估具有超越跟踪退化和改进的广泛好处。其累积价值容易被忽视，因为成本在前可见而利益随后积累。

## 如何评估 AI 智能体

我们看到几种常见的智能体类型在当今大规模部署，包括代码智能体、研究智能体、计算机使用智能体和对话智能体。每种类型可能部署在广泛的行业中，但可以使用相似的技术进行评估。你不需要从头发明评估。以下部分描述了几种智能体类型的经过验证的技术。使用这些方法作为基础，然后扩展到你的领域。

### 智能体评分器的类型

智能体评估通常结合三种类型的评分器：基于代码的、基于模型的和人工的。每个评分器评估转录记录或结果的某些部分。有效评估设计的基本组件是为工作选择合适的评分器。

#### 基于代码的评分器

| 方法 | 优势 | 劣势 |
|---------|-----------|------------|
| 字符串匹配检查（精确、正则表达式、模糊等） | 快速 | 对不完全匹配预期模式的有效变化很脆弱 |
| 二元测试（失败到通过、通过到通过） | 便宜 | 缺乏细微差别 |
| 静态分析（lint、类型、安全） | 客观 | 对于评估一些更主观的任务有限 |
| 结果验证 | 可重现 | |
| 工具调用验证（使用的工具、参数） | 易于调试 | |
| 转录记录分析（所用轮数、令牌使用） | 验证特定条件 | |

#### 基于模型的评分器

| 方法 | 优势 | 劣势 |
|---------|-----------|------------|
| 基于量规的评分 | 灵活 | 非确定性 |
| 自然语言断言 | 可扩展 | 比代码更昂贵 |
| 成对比较 | 捕捉细微差别 | 需要与人工评分器校准以获得准确性 |
| 基于参考的评估 | 处理开放式任务 | |
| 多裁判共识 | 处理自由格式输出 | |

#### 人工评分器

| 方法 | 优势 | 劣势 |
|---------|-----------|------------|
| 主题专家审查 | 黄金标准质量 | 昂贵 |
| 众包判断 | 匹配专家用户判断 | 慢 |
| 抽查采样 | 用于校准基于模型的评分器 | 通常需要大规模访问人类专家 |
| A/B 测试 | | |
| 标注者间一致性 | | |

对于每个任务，评分可以是加权的（组合评分器分数必须达到阈值）、二元的（所有评分器必须通过）或混合的。

### 能力与回归评估

**能力**或"质量"评估问："这个智能体能做好什么？"它们应该从低通过率开始，针对智能体 struggling 的任务，为团队提供一个要攀登的山丘。

**回归**评估问："智能体是否仍然处理它过去处理的所有任务？"并应该有近 100% 的通过率。它们防止倒退，因为分数下降表明某些东西坏了需要改进。当团队在能力评估上攀登时，同时运行回归评估以确保更改不会在其他地方引起问题也很重要。

在智能体发布和优化后，高通过率的能力评估可以"毕业"成为连续运行的回归套件，以捕捉任何漂移。曾经测量"我们能否做到这一点？"的任务现在测量"我们是否能可靠地继续做到这一点？"

### 评估代码智能体

代码智能体编写、测试和调试代码，导航代码库并运行命令，就像人类开发者一样。现代代码智能体的有效评估通常依赖于规范明确的任务、稳定的测试环境，以及对生成代码的彻底测试。

确定性评分器对代码智能体来说是自然的，因为软件通常很容易评估：代码是否运行，测试是否通过？两个广泛使用的代码智能体基准，SWE-bench Verified 和 Terminal-Bench，遵循这种方法。SWE-bench Verified 给智能体来自流行 Python 存储库的 GitHub 问题，并通过运行测试套件来评分解决方案；只有当解决方案修复了失败的测试而不破坏现有测试时，它才算通过。LLM 在这一评估中在短短一年内从 40% 进步到>80%。Terminal-Bench 采取不同的轨道：它测试端到端的技术任务，例如从源代码构建 Linux 内核或训练 ML 模型。

一旦你有一组通过或失败的测试来验证代码任务的关键结果，通常也有用评分转录记录。例如，基于启发式的代码质量规则可以根据超越通过测试的方面评估生成的代码，具有清晰量规的基于模型的评分器可以评估行为，如智能体如何调用工具或与用户交互。

**示例：代码智能体的理论评估**

考虑一个代码任务，智能体必须修复身份验证绕过漏洞。如下所示的说明性 YAML 文件中，可以使用评分器和指标来评估这个智能体。

```yaml
task:
  id: "fix-auth-bypass_1"
  desc: "修复密码字段为空时的身份验证绕过..."
  graders:
    - type: deterministic_tests
      required: [test_empty_pw_rejected.py, test_null_pw_rejected.py]
    - type: llm_rubric
      rubric: prompts/code_quality.md
    - type: static_analysis
      commands: [ruff, mypy, bandit]
    - type: state_check
      expect:
        security_logs: {event_type: "auth_blocked"}
    - type: tool_calls
      required:
        - {tool: read_file, params: {path: "src/auth/*"}}
        - {tool: edit_file}
        - {tool: run_tests}
  tracked_metrics:
    - type: transcript
      metrics:
        - n_turns
        - n_toolcalls
        - n_total_tokens
    - type: latency
      metrics:
        - time_to_first_token
        - output_tokens_per_sec
        - time_to_last_token
```

注意这个示例展示可用评分器的完整范围以供说明。在实践中，代码评估通常依赖单元测试进行正确性验证和 LLM 量规进行整体代码质量评估，仅在需要时添加额外的评分器和指标。

### 评估对话智能体

对话智能体在支持、销售或教练等领域与用户交互。与传统聊天机器人不同，它们维持状态、使用工具，并在对话中途采取行动。虽然代码和研究智能体也可以涉及与用户的多轮交互，但对话智能体呈现独特的挑战：交互本身的质量是你评估的一部分。对话智能体的有效评估通常依赖于可验证的结束状态结果和捕捉任务完成和交互质量的量规。与大多数其他评估不同，它们通常需要第二个 LLM 来模拟用户。我们在对齐审计智能体中使用这种方法，通过扩展的对抗性对话压力测试模型。

对话智能体的成功可以是多维的：工单是否解决（状态检查），是否在<10 轮内完成（转录记录约束），语气是否合适（LLM 量规）？两个包含多维性的基准是τ-Bench 及其继任者τ2-Bench。这些模拟跨零售支持和航班预订等领域的多轮交互，其中一个模型扮演用户角色，而智能体导航现实场景。

**示例：对话智能体的理论评估**

考虑一个支持任务，智能体必须处理沮丧客户的退款。

```yaml
graders:
  - type: llm_rubric
    rubric: prompts/support_quality.md
    assertions:
      - "智能体对客户沮丧表现出同理心"
      - "解决方案被清楚解释"
      - "智能体响应基于 fetch_policy 工具结果"
  - type: state_check
    expect:
      tickets: {status: resolved}
      refunds: {status: processed}
  - type: tool_calls
    required:
      - {tool: verify_identity}
      - {tool: process_refund, params: {amount: "<=100"}}
      - {tool: send_confirmation}
  - type: transcript
    max_turns: 10
tracked_metrics:
  - type: transcript
    metrics:
      - n_turns
      - n_toolcalls
      - n_total_tokens
  - type: latency
    metrics:
      - time_to_first_token
      - output_tokens_per_sec
      - time_to_last_token
```

如我们的代码智能体示例，此任务展示多种评分器类型以供说明。在实践中，对话智能体评估通常使用基于模型的评分器来评估沟通质量和目标完成，因为许多任务——如回答问题——可能有多个"正确"解决方案。

### 评估研究智能体

研究智能体收集、综合和分析信息，然后产生输出如答案或报告。与代码智能体不同，单元测试提供二元通过/失败信号，研究质量只能相对任务来判断。什么算"全面"、"来源良好"甚至"正确"取决于上下文：市场扫描、收购尽职调查和科学报告各需要不同标准。

研究评估面临独特挑战：专家可能对综合是否全面有分歧，随着参考内容不断变化的地面真相会变化，更长、更开放的输出创造更多犯错空间。例如，BrowseComp 这样的基准测试 AI 智能体是否能在开放网络中找到干草堆里的针——设计为容易验证但难以解决的问题。

构建研究智能体评估的一种策略是结合评分器类型。真实性检查验证声明有检索来源支持，覆盖率检查定义好答案必须包括的关键事实，来源质量检查确认咨询的来源是权威的，而不仅仅是第一个检索的。对于有客观正确答案的任务（"公司 X 的第三季度收入是多少？"），精确匹配有效。LLM 可以标记无支持的声明和覆盖缺口，但也验证开放式综合的连贯性和完整性。

鉴于研究质量的主观性质，基于 LLM 的量规应经常针对专家人类判断校准以有效评估这些智能体。

### 计算机使用智能体

计算机使用智能体通过人类相同的接口与软件交互——截图、鼠标点击、键盘输入和滚动——而不是通过 API 或代码执行。它们可以使用任何具有图形用户界面（GUI）的应用程序，从设计工具到遗留企业软件。评估需要在真实或沙盒环境中运行智能体，它可以在其中使用软件应用程序，并检查是否达到预期结果。例如，WebArena 测试基于浏览器的任务，使用 URL 和页面状态检查验证智能体导航正确，以及对修改数据的任务进行后端状态验证（确认订单实际下而不是只是确认页面出现）。OSWorld 将其扩展到完整的操作系统控制，评估脚本在任务完成后检查各种工件：文件系统状态、应用程序配置、数据库内容和 UI 元素属性。

浏览器使用智能体需要在令牌效率和延迟之间平衡。基于 DOM 的交互执行快速但消耗许多令牌，而基于截图的交互更慢但更令牌高效。例如，当要求 Claude 总结维基百科时，从 DOM 提取文本更高效。当在 Amazon 找新笔记本电脑时，截图更高效（因为提取整个 DOM 是令牌密集的）。在我们的 Claude for Chrome 产品中，我们开发了评估来检查智能体是否为每个上下文选择合适的工具。这使我们能够更快更准确地完成基于浏览器的任务。

### 如何思考智能体评估中的非确定性

无论智能体类型，智能体行为在运行之间变化，这使得评估结果比第一眼看起来更难解释。每个任务有自己的成功率——可能一个任务 90%，另一个 50%——一个在一次评估运行中通过的任务可能在下次失败。有时，我们想要测量的是智能体对任务成功有多频繁（试验的比例）。

两个指标帮助捕捉这种细微差别：

**pass@k** 测量智能体在 k 次尝试中获得至少一个正确解决方案的可能性。随着 k 增加，pass@k 分数上升：更多"射门"意味着至少 1 次成功的更高几率。50% pass@1 分数意味着模型在评估中一半任务的第一次尝试成功。在代码中，我们通常最感兴趣智能体第一次尝试找到解决方案——pass@1。在其他情况下，提出许多解决方案是有效的，只要一个工作。

**pass^k** 测量所有 k 次试验成功的概率。随着 k 增加，pass^k 下降，因为要求跨更多试验的一致性是一个更难清除的障碍。如果你的智能体有 75% 每次试验成功率并运行 3 次试验，通过所有三次的概率是 (0.75)^3 = 42%。这个指标对客户面向的智能体特别重要，用户期望每次可靠行为。

pass@k 和 pass^k 随试验增加而发散。在 k=1，它们相同（都等于每次试验成功率）。到 k=10，它们讲述相反的故事：pass@k 接近 100% 而 pass^k 降到 0%。

两个指标都有用，使用哪个取决于产品要求：pass@k 用于一个成功重要的工具，pass^k 用于一致性本质的智能体。

## 从零到一：智能体优秀评估路线图

本节概述我们实用的、经过实地测试的建议，从无评估到你信任的评估。将此视为评估驱动智能体开发的路线图：早期定义成功，清楚测量它，并持续迭代。

### 收集初始评估数据集的任务

**步骤 0. 早期开始**

我们看到团队延迟构建评估，因为他们认为需要数百个任务。实际上，20-50 个来自真实失败的简单任务是极好的开始。毕竟，在早期智能体开发中，对系统的每次更改通常有清晰、明显的影向，这种大效应大小意味着小样本大小足够。更成熟的智能体可能需要更大、更困难的评估来检测更小效应，但最好一开始采用 80/20 方法。评估等待越久越难构建。早期，产品要求自然地转化为测试用例。等待太久，你从实时系统逆向工程成功标准。

**步骤 1. 从你已经手动测试的开始**

从你在开发期间运行的手动检查开始——每次发布前验证的行为和最终用户尝试的常见任务。如果你已经在生产中，查看你的 bug 跟踪器和支持队列。将用户报告的失败转换为测试用例确保你的套件反映实际使用；按用户影响优先帮助你投资在计算的地方。

**步骤 2：编写无歧义任务并附带参考解决方案**

正确获取任务质量比看起来更难。好任务是两个领域专家会独立到达相同通过/失败判决的任务。他们能自己通过任务吗？如果不能，任务需要细化。任务规范中的歧义成为指标中的噪声。同样的适用于基于模型评分器的标准：模糊量规产生不一致判断。

每个任务应该由正确遵循指令的智能体通过。这可能很微妙。例如，审查 Terminal-Bench 揭示如果任务要求智能体编写脚本但不指定文件路径，而测试假设脚本的特定文件路径，智能体可能失败并非它自己的错。评分器检查的一切应该从任务描述清楚；智能体不应因模糊规范失败。对于前沿模型，跨许多试验 0% 通过率（即 0% pass@100）通常是最常是任务损坏的信号，而非无能智能体，是检查你的任务规范和评分器的标志。对于每个任务，创建参考解决方案很有用：已知通过所有评分器的工作输出。这证明任务可解并验证评分器正确配置。

**步骤 3：构建平衡问题集**

测试行为应该发生和不应该发生的情况。单面评估创造单面优化。例如，如果你只测试智能体应该在什么时候搜索，你可能得到几乎搜索一切的智能体。尽量避免类别不平衡评估。我们在为 Claude.ai 的网页搜索构建评估时亲身学到这个。挑战是防止模型在不应该时搜索，同时保留它在适当时做广泛研究的能力。团队构建覆盖两个方向的评估：模型应该搜索的查询（如找天气）和应该从现有知识回答的查询（如"谁创立了 Apple？"）。在触发不足（应该在时不搜索）和过度触发（不应该时搜索）之间打击正确平衡很困难，需要对提示和评估的多轮细化。随着更多示例问题出现，我们继续添加到评估以改进覆盖。

### 设计评估框架和评分器

**步骤 4：构建具有稳定环境的稳健评估框架**

评估中的智能体功能大致相同于生产中使用的智能体，环境本身不引入更多噪声，这是基本的。每次试验应该从干净环境开始"隔离"。运行之间不必要的共享状态（遗留文件、缓存数据、资源耗尽）可能由于基础设施不稳定性而非智能体性能导致相关失败。共享状态也可能人为夸大性能。例如，在一些内部评估中我们观察到 Claude 通过检查先前试验的 git 历史在某些任务上获得不公平优势。如果多个不同试验因为环境的相同限制（如有限 CPU 内存）失败，这些试验不独立因为它们受相同因素影响，评估结果对测量智能体性能变得不可靠。

**步骤 5：深思熟虑地设计评分器**

如上讨论，优秀评估设计涉及为智能体和任务选择最佳评分器。我们推荐在可能时选择确定性评分器，在需要或额外灵活性时使用 LLM 评分器，并明智使用人工评分器进行额外验证。

有一个常见直觉检查智能体是否遵循非常具体步骤如正确顺序的工具调用序列。我们发现这种方法太僵硬，导致过度脆弱测试，因为智能体经常找到评估设计师未预料到的有效方法。为了不不必要惩罚创造力，通常更好评估智能体产生的东西，不是它采取的路径。

对于有多个组件的任务，构建部分信用。正确识别问题并验证客户但未能处理退款的支智能体比立即失败的智能体有意义地更好。在结果中表示这种成功连续体很重要。

模型评分通常需要仔细迭代以验证准确性。LLM-as-judge 评分器应与人类专家紧密校准，以获得人类评分和模型评分之间分歧很小的信心。为避免幻觉，给 LLM 退出方式，如提供返回"未知"的指令当它没有足够信息时。创建清晰、结构化量规来评分任务每个维度，然后用隔离的 LLM-as-judge 评分每个维度，而不是用一个评分所有维度，也可能有帮助。一旦系统稳健，仅偶尔使用人工审查就足够。

一些评估有微妙失败模式，即使智能体性能良好也导致低分，因为智能体因评分器 bug、智能体框架约束或歧义未能解决问题。即使复杂团队也可能错过这些问题。例如，Opus 4.1.3 最初在 CORE-Bench 上得分 42%，直到 Anthropic 研究员发现多个问题：僵化评分惩罚"96.12"当期望"96.124991..."，模糊任务规范，以及无法精确重现的随机任务。修复 bug 并使用更少约束的支架后，Opus 4.1.3 分数跳到 95%。同样，METR 在他们的时间范围基准中发现几个配置错误的任务，要求智能体优化到陈述分数阈值，但评分要求超过该阈值。这惩罚遵循指令的模型如 Claude，而忽略陈述目标的模型获得更好分数。仔细双重检查任务和评分器可以帮助避免这些问题。

使你的评分器抵抗绕过或黑客。智能体不应能轻易"作弊"评估。任务和评分器应该设计为使通过真正要求解决问题而不是利用意外漏洞。

### 长期维护和使用评估

**步骤 6：检查转录记录**

除非你阅读许多试验的转录记录和分数，否则你不知道评分器是否工作良好。在 Anthropic，我们投资于查看评估转录记录的工具，我们定期花时间阅读它们。当任务失败时，转录记录告诉你智能体是否犯真实错误，或你的评分器拒绝有效解决方案。它也经常表面智能体和评估行为的关键细节。

失败应该显得公平：清楚智能体错在哪里和为什么。当分数不爬升时，我们需要信心这是智能体性能而非评估。阅读转录记录是你验证评估测量实际重要的东西的方式，是智能体开发的关键技能。

**步骤 7：监控能力评估饱和**

100% 的评估跟踪回归但不提供改进信号。评估饱和发生在智能体通过所有可解任务，不留改进空间时。例如，SWE-Bench Verified 分数今年从 30% 开始，前沿模型现在接近>80% 饱和。当评估接近饱和，进步也会变慢，因为只有最困难任务剩余。这可使结果欺骗，因为大能力改进表现为分数小增加。例如，代码审查初创 Qodo 最初对 Opus 4.1.3 印象不深，因为他们的一次性代码评估没有捕捉更长、更复杂任务的增益。作为响应，他们开发了新的智能体评估框架，提供进步更清晰图景。

作为规则，我们不会面价值接受评估分数，直到有人深入研究评估细节并阅读一些转录记录。如果评分不公平、任务歧义、有效解决方案受罚或框架约束模型，评估应该修订。

**步骤 8：通过开放贡献和维护长期保持评估套件健康**

评估套件是需要持续关注和平所有权以保持有用的活工件。

在 Anthropic，我们实验各种评估维护方法。证明最有效的是建立专用评估团队拥有核心基础设施，而领域专家和产品团队贡献大多数评估任务并自己运行评估。

对于 AI 产品团队，拥有和迭代评估应该像维护单元测试一样常规。团队可能在"工作"的 AI 功能上浪费数周，但未能满足设计良好的评估会早期表面的未陈述期望。定义评估任务是压力测试产品要求是否具体到开始构建的最佳方式之一。

我们推荐实践评估驱动开发：构建评估定义计划能力在智能体能实现之前，然后迭代直到智能体表现良好。内部，我们经常构建今天"足够好"的功能，但是对几个月后模型能做什么的赌注。从低通过率开始的能力评估使这可见。当新模型下降，运行套件快速揭示哪些赌注回报。

最接近产品要求和用户的人最有资格定义成功。用当前模型能力，产品经理、客户成功经理或销售人员可以使用 Claude Code 作为 PR 贡献评估任务——让他们！或者，甚至更好，主动启用他们。

## 评估如何与其他方法配合全面理解智能体

自动评估可以在不部署到生产或影响真实用户的情况下对智能体运行数千任务。但这只是理解智能体性能的多种方式之一。完整图景包括生产监控、用户反馈、A/B 测试、手动转录记录审查和系统人工评估。

**理解 AI 智能体性能方法概述**

| 方法 | 优点 | 缺点 |
|--------|------|------|
| **自动评估** | 更快迭代 | 需要更多前期投资构建 |
| （无需真实用户程序化运行测试） | 完全可重现 | 需要持续维护随产品和模型演变以避免漂移 |
| | 无用户影响 | 如果不匹配真实使用模式可能创造虚假信心 |
| | 可在每次提交运行 | |
| | 测试场景大规模无需生产部署 | |
| **生产监控** | 揭示大规模真实用户行为 | 被动；问题在你知道前到达用户 |
| （跟踪实时系统指标和错误） | 捕捉合成评估错过的问题 | 信号可能嘈杂 |
| | 提供智能体实际表现地面真相 | 需要投资仪器化 |
| | | 缺乏评分地面真相 |
| **A/B 测试** | 测量实际用户结果（保留、任务完成） | 慢；需要数天或数周达到显著性并需要足够流量 |
| （用真实用户流量比较变体） | 控制混淆因素 | 只测试你部署的更改 |
| | 可扩展和系统化 | 如果没有彻底审查转录记录，对指标变化底层"为什么"信号较少 |
| **用户反馈** | 表面你未预料的问题 | 稀疏和自我选择 |
| （明确信号如向下拇指或 bug 报告） | 带来自真实人类用户的真实示例 | 偏向严重问题 |
| | 反馈经常与产品目标相关 | 用户很少解释为什么某些东西失败 |
| | | 不自动化 |
| | | 主要依赖用户发现问题可能有负面用户影响 |
| **手动转录记录审查** | 建立失败模式直觉 | 时间密集 |
| （人类阅读智能体对话） | 捕捉自动检查错过微妙质量问题 | 不扩展 |
| | 帮助校准"好"看起来像什么并掌握细节 | 覆盖不一致 |
| | | 审查者疲劳或不同审查者可能影响信号质量 |
| | | 通常仅给出定性信号而非清晰定量评分 |
| **系统人工研究** | 来自多个人类评分者的黄金标准质量判断 | 相对昂贵和慢周转 |
| （训练评分者结构化合评分智能体输出） | 处理主观或歧义任务 | 难以频繁运行 |
| | 提供改进基于模型评分器的信号 | 评分者间分歧需要调和 |
| | | 复杂领域（法律、金融、医疗）需要人类专家进行研究 |

这些方法映射到智能体开发的不同阶段。自动评估在发布前和 CI/CD 中特别有用，在每次智能体更改和模型升级时运行，作为对抗质量问题的第一道防线。生产监控在发布后启动，检测分布漂移和未预料的现实失败。A/B 测试在你有足够流量时验证重大更改。用户反馈和转录记录审查是持续实践以填补差距：不断分类反馈，每周采样转录记录阅读，并按需深入挖掘。保留系统人工研究校准 LLM 评分器或评估人类共识作为参考标准的主观输出。

像安全工程的瑞士奶酪模型，没有单一评估层捕捉每个问题。多种方法结合，通过一层的失败被另一层捕捉。

最有效团队结合这些方法：自动评估快速迭代，生产监控地面真相，定期人工审查校准。

## 结论

没有评估的团队陷入被动循环——修复一个失败，创造另一个，无法区分真实回归从噪声。早期投资的团队发现相反：开发加速，失败成为测试用例，测试用例防止回归，指标取代猜测。评估给团队清晰山丘攀登，将"智能体感觉更差"转化为可操作的东西。价值累积，但仅当你把评估作为核心组件而非事后思考。

模式因智能体类型而异，但这里描述的基础不变。早期开始，不要等待完美套件。从你看到的失败获取现实任务。定义无歧义、稳健成功标准。深思熟虑设计评分器并组合多种类型。确保问题对模型足够难。迭代评估改进信噪比。阅读转录记录！

AI 智能体评估仍然是新生、快速演变的领域。随着智能体承担更长任务，在多智能体系统协作，处理越来越主观工作，我们需要适应技术。随着我们学习更多，我们会继续分享最佳实践。

## 附录：评估框架

几个开源和商业框架可以帮助团队实施智能体评估而无需从头构建基础设施。正确选择取决于你的智能体类型、现有堆栈，以及你是否需要离线评估、生产可观测性或两者。

**Harbor** 设计用于在容器化环境中运行智能体，有跨云提供商大规模运行试验的基础设施，以及定义任务和评分器的标准化格式。流行基准如 Terminal-Bench 2.0 通过 Harbor 注册表发布，使运行建立基准和自定义评估套件变得容易。

**Braintrust** 是结合离线评估与生产可观测性和实验跟踪的平台——对需要在开发期间迭代和生产中监控质量的团队有用。其 `autoevals` 库包括为事实性、相关性和其他常见维度预构建的评分器。

**LangSmith** 提供追踪、离线和在线评估，以及与 LangChain 生态系统紧密集成的数据集管理。

**Langfuse** 作为自托管开源替代方案提供类似功能，适用于有数据驻留要求团队。

**Arize** 提供 Phoenix，开源 LLM 追踪、调试和离线或在线评估平台，以及 AX，扩展 Phoenix 用于规模、优化和监控的 SaaS 产品。

许多团队组合多种工具，自己卷评估框架，或仅使用简单评估脚本作为起点。我们发现框架可以是加速进步和标准化的宝贵方式，但它们仅和你通过它们运行的评估任务一样好。通常最好快速选择适合工作流的框架，然后投资能量在评估本身，通过迭代高质量测试用例和评分器。
