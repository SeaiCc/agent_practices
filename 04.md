# 为智能体编写高效的工具——使用智能体

来源：https://www.anthropic.com/engineering/writing-tools-for-agents

Anthropic 工程部
**发布日期：2025 年 9 月 11 日**

智能体的效果取决于我们提供的工具。我们分享如何编写高质量的工具和评估，以及如何通过使用 Claude 来优化其自身工具以提升性能。

模型上下文协议（MCP）可以为 LLM 智能体提供数百个工具来解决现实世界任务。但我们如何使这些工具发挥最大效果？

在本文中，我们描述了在各种智能体 AI 系统中提高性能的最有效技术。

我们将首先介绍如何：

- 构建和测试工具原型
- 创建并运行与智能体一起对工具进行全面评估
- 与 Claude Code 等智能体协作，自动提高工具性能

最后，我们总结在过程中发现的关键原则：

- 选择正确的工具实现（以及不实现）
- 工具命名空间以明确功能边界
- 从工具向智能体返回有意义的上下文
- 优化工具响应以提高令牌效率
- 对工具描述和规范进行提示工程

## 什么是工具？

在计算领域，确定性系统每次给定相同输入时产生相同输出，而*非确定性*系统——如智能体——即使在相同起始条件下也能生成不同的响应。

当我们传统地编写软件时，我们在建立确定性系统之间的契约。例如，像 `getWeather("NYC")` 这样的函数调用每次调用时总是以完全相同的方式获取纽约市的天气。

工具是一种新型软件，反映了确定性系统与非确定性智能体之间的契约。当用户问"我今天应该带伞吗？"时，智能体可能调用天气工具，从通用知识回答，或者甚至先问一个关于位置的澄清问题。偶尔，智能体可能会产生幻觉或无法理解如何使用工具。

这意味着在为智能体编写软件时，我们需要从根本上重新思考我们的方法：我们不是像为其他开发者或系统编写函数和 API 那样编写工具和 MCP 服务器，而是需要为智能体设计它们。

我们的目标是通过使用工具追求各种成功策略，增加智能体能够有效解决广泛任务的表面积。幸运的是，根据我们的经验，对智能体最"符合人体工程学"的工具最终对人类来说也出奇地直观易懂。

## 如何编写工具

在本节中，我们描述如何与智能体协作来编写和改进你提供给它们的工具。首先快速搭建工具原型并在本地测试。然后运行全面评估以衡量后续变化。与智能体一起工作，你可以重复评估和改进工具的过程，直到你的智能体在现实世界任务上实现强劲性能。

### 构建原型

在没有亲自动手之前，很难预见哪些工具智能体会觉得符合人体工程学，哪些不会。首先快速搭建工具原型。如果你使用 Claude Code 编写工具（可能一次完成），最好为工具依赖的任何软件库、API 或 SDK（包括可能的 MCP SDK）提供文档。LLM 友好的文档通常可以在官方文档站点的扁平 `llms.txt` 文件中找到（这是我们 API 的）。

将工具包装在本地 MCP 服务器或桌面扩展（DXT）中将允许你在 Claude Code 或 Claude Desktop 应用中连接和测试工具。

要将本地 MCP 服务器连接到 Claude Code，运行 `claude mcp add <name> <command> [args...]`。

要将本地 MCP 服务器或 DXT 连接到 Claude Desktop 应用，分别导航到 `Settings > Developer` 或 `Settings > Extensions`。

工具也可以直接传递到 Anthropic API 调用中进行编程测试。

自己测试工具以识别任何粗糙边缘。收集用户反馈以建立对你期望工具启用的用例和提示的直觉。

### 运行评估

接下来，你需要通过运行评估来衡量 Claude 使用工具的效果。首先生成大量评估任务，基于现实世界使用。我们建议与智能体协作以帮助分析结果并确定如何改进工具。在我们的工具评估食谱中查看端到端过程。

**生成评估任务**

使用早期原型，Claude Code 可以快速探索工具并创建数十个提示和响应对。提示应该受现实世界使用启发，基于真实数据源和服务（例如，内部知识库和微服务）。我们建议你避免过于简单或表面的"沙盒"环境，这些环境不足以充分压力测试你的工具。强大的评估任务可能需要多次工具调用——可能数十次。

以下是一些强大任务的示例：

- 安排下周与 Jane 开会讨论我们最新的 Acme Corp 项目。附上上次项目规划会议的笔记并预订会议室。
- 客户 ID 9182 报告单次购买尝试被收费三次。找到所有相关日志条目并确定是否有其他客户受到相同问题影响。
- 客户 Sarah Chen 刚刚提交了取消请求。准备挽留优惠。确定：(1) 他们为何离开，(2) 什么挽留优惠最具吸引力，(3) 在提供优惠前我们应该注意的任何风险因素。

以下是一些较弱任务：

- 下周与 jane@acme.corp 安排会议。
- 在支付日志中搜索 `purchase_complete` 和 `customer_id=9182`。
- 按客户 ID 45892 查找取消请求。

每个评估提示应该与可验证的响应或结果配对。你的验证器可以简单如真实值和采样响应之间的精确字符串比较，或复杂如召集 Claude 来判断响应。避免过于严格的验证器，因格式化、标点或有效替代措辞等无关差异而拒绝正确响应。

对于每个提示 - 响应对，你可以选择性地指定你期望智能体在解决问题时调用的工具，以衡量智能体在评估期间是否成功理解每个工具的目的。但是，因为可能有多个有效路径正确解决问题，尽量避免过度指定或过度拟合策略。

**运行评估**

我们建议使用直接 LLM API 调用编程运行评估。使用简单智能体循环（包装交替 LLM API 和工具调用的 `while` 循环）：每个评估任务一个循环。每个评估智能体应获得单个任务提示和你的工具。

在评估智能体的系统提示中，我们建议指示智能体输出不仅结构化响应块（用于验证），还包括推理和反馈块。指示智能体在工具调用和响应块*之前*输出这些可能通过触发思维链（CoT）行为增加 LLM 的有效智能。

如果你使用 Claude 运行评估，你可以打开交错思考以获得类似功能"开箱即用"。这将帮助你探测智能体为何调用或不调用某些工具，并突出工具描述和规范中需要改进的具体领域。

除了顶级准确率，我们建议收集其他指标，如单个工具调用和任务的总运行时间、工具调用总数、总令牌消耗和工具错误。跟踪工具调用可以帮助揭示智能体追求的共同工作流，并为工具整合提供一些机会。

**分析结果**  
智能体是你发现问题和提供反馈的有益伙伴，从矛盾工具描述到低效工具实现和混乱工具模式。但是，请记住智能体在反馈和响应中省略的内容往往比包含的内容更重要。LLM 并不总是言出即所思。

观察智能体在哪里卡住或困惑。阅读评估智能体的推理和反馈（或 CoT）以识别粗糙边缘。回顾原始转录（包括工具调用和工具响应）以捕获智能体 CoT 中未明确描述的任何行为。阅读字里行间；记住你的评估智能体不一定知道正确答案和策略。

分析工具调用指标。大量冗余工具调用可能表明需要调整分页或令牌限制参数；大量无效参数工具错误可能表明工具可以使用更清晰描述或更好示例。当我们推出 Claude 的网络搜索工具时，我们发现 Claude 不必要地在工具的 `query` 参数后附加 `2025`，偏置搜索结果并降低性能（我们通过改进工具描述将 Claude 引向正确方向）。

### 与智能体协作

你甚至可以让智能体分析结果并为你改进工具。只需连接评估智能体的转录并粘贴到 Claude Code。Claude 是分析转录和一次性重构大量工具的专家——例如，确保工具实现和描述在新更改时保持自洽。

事实上，本文中大部分建议来自使用 Claude Code 反复优化内部工具实现。我们的评估建立在内部工作区之上，镜像内部工作流的复杂性，包括真实项目、文档和消息。

我们依赖保留测试集以确保不过度拟合"训练"评估。这些测试集显示我们可以提取额外性能改进，甚至超过我们使用"专家"工具实现所实现的——无论这些工具是由研究人员手动编写还是由 Claude 本身生成。

在下一节中，我们将分享从这一过程中学到的一些内容。

## 编写高效工具的原则

在本节中，我们将学习提炼为编写高效工具的几条指导原则。

### 为智能体选择正确的工具

更多工具并不总是带来更好结果。我们观察到一个常见错误是工具仅仅包装现有软件功能或 API 端点——无论工具是否适合智能体。这是因为智能体对传统软件有不同的"可供性"——也就是说，它们有不同的方式来感知可以用这些工具采取的潜在行动。

LLM 智能体有有限的"上下文"（即，它们一次可以处理的信息量有限），而计算机内存便宜且丰富。考虑在通讯录中搜索联系人的任务。传统软件程序可以有效地存储和处理联系人列表一次一个，在继续之前检查每一个。

但是，如果 LLM 智能体使用返回所有联系人的工具，然后必须逐个令牌阅读每一个，它正在有限的上下文空间上浪费在无关信息上（想象通过从顶部到底部阅读每一页在通讯录中搜索联系人——即，通过暴力搜索）。更好和更自然的方法（对智能体和人类都如此）是首先跳转到相关页面（可能按字母顺序找到它）。

我们建议构建一些深思熟虑的工具，针对特定高影响工作流，与评估任务匹配并从那里扩展。在通讯录案例中，你可能选择实现 `search_contacts` 或 `message_contact` 工具而不是 `list_contacts` 工具。

工具可以整合功能，在底层处理可能*多个*离散操作（或 API 调用）。例如，工具可以用相关元数据丰富工具响应，或在单个工具调用中处理频繁链接的多步骤任务。

以下是一些示例：

- 不实现 `list_users`、`list_events` 和 `create_event` 工具，考虑实现 `schedule_event` 工具，它查找可用性并安排事件。
- 不实现 `read_logs` 工具，考虑实现 `search_logs` 工具，它只返回相关日志行和一些周围上下文。
- 不实现 `get_customer_by_id`、`list_transactions` 和 `list_notes` 工具，实现 `get_customer_context` 工具，它一次性编译客户的所有最近和相关信息。

确保你构建的每个工具有清晰、独特的目的。工具应该使智能体能够像人类一样细分和解决问题，给定访问相同底层资源，同时减少本来会被中间输出消耗的上下文。

太多工具或重叠工具也可能分散智能体追求高效策略的注意力。仔细、有选择地规划你构建（或不构建）的工具真的会带来回报。

### 为工具设置命名空间

你的 AI 智能体可能获得数十个 MCP 服务器和数百个不同工具的访问——包括其他开发者的。当工具在功能上重叠或目的模糊时，智能体可能对使用哪个感到困惑。

命名空间（将相关工具分组在共同前缀下）可以帮助 delineate 大量工具之间的边界；MCP 客户端有时默认这样做。例如，按服务（例如，`asana_search`、`jira_search`）和按资源（例如，`asana_projects_search`、`asana_users_search`）为工具设置命名空间，可以帮助智能体在正确时间选择正确工具。

我们发现选择基于前缀和后缀的命名空间对我们的工具使用评估有非琐碎影响。影响因 LLM 而异，我们鼓励你根据自己的评估选择命名方案。

智能体可能调用错误工具，以错误参数调用正确工具，调用太少工具，或错误处理工具响应。通过有选择地实现名称反映任务自然细分的工具，你同时减少加载到智能体上下文的工具数量和工具描述，并将智能体计算从智能体上下文卸载回工具调用本身。这减少智能体犯错的总体风险。

### 从工具返回有意义的上下文

同样，工具实现应该小心只返回高信号信息回智能体。它们应该优先考虑上下文相关性而非灵活性，并避免低级技术标识符（例如：`uuid`、`256px_image_url`、`mime_type`）。像 `name`、`image_url` 和 `file_type` 这样的字段更可能直接告知智能体的下游行动和响应。

智能体也往往比处理 cryptic 标识符更成功地处理自然语言名称、术语或标识符。我们发现仅仅将任意字母数字 UUID 解析为更具语义意义和可解释的语言（甚至 0 索引 ID 方案）通过减少幻觉显著改善 Claude 在检索任务中的精度。

在某些情况下，智能体可能需要与两者交互的灵活性，自然语言和技术标识符输出，如果只是为了触发下游工具调用（例如，`search_user(name='jane')` → `send_message(id=12345)`）。你可以在工具中暴露简单的 `response_format` 枚举参数使两者都可用，允许智能体控制工具返回 `"concise"` 或 `"detailed"` 响应。

你可以添加更多格式以获得更大灵活性，类似于 GraphQL，你可以选择确切接收哪些信息。以下是控制工具响应冗长度 ResponseFormat 枚举的示例：

```
enum ResponseFormat {
   DETAILED = "detailed",
   CONCISE = "concise"
}
```

Slack 线程和线程回复由唯一 `thread_ts` 标识，这是获取线程回复所必需的。`thread_ts` 和其他 ID（`channel_id`、`user_id`）可以从 `"detailed"` 工具响应中检索以启用需要这些的进一步工具调用。`"concise"` 工具响应只返回线程内容并排除 ID。在此示例中，我们使用约 1/3 的令牌与 `"concise"` 工具响应。

甚至你的工具响应结构——例如 XML、JSON 或 Markdown——都可能影响评估性能：没有一刀切的解决方案。这是因为 LLM 在下一个令牌预测上训练，往往在与训练数据匹配的格式上表现更好。最佳响应结构将因任务和智能体而异。我们鼓励你根据自己的评估选择最佳响应结构。

### 优化工具响应以提高令牌效率

优化上下文质量很重要。但优化工具响应中返回给智能体的上下文*数量*也很重要。

我们建议对任何可能消耗大量上下文的工具响应实施分页、范围选择、过滤和/或截断的组合，带有合理的默认参数值。对于 Claude Code，我们默认将工具响应限制为 25,000 令牌。我们期望智能体的有效上下文长度随时间增长，但对上下文高效工具的需求将保持不变。

如果你选择截断响应，确保用有帮助的指示引导智能体。你可以直接鼓励智能体追求更令牌高效的策略，如进行许多小而有针对性的搜索而不是单个广泛搜索用于知识检索任务。同样，如果工具调用引发错误（例如，在输入验证期间），你可以提示工程错误响应以清晰传达具体和可操作的改进，而不是不透明错误代码或跟踪。

工具截断和错误响应可以引导智能体朝向更令牌高效的工具使用行为（使用过滤器或分页）或给出正确格式工具输入的示例。

### 对工具描述进行提示工程

我们现在来到改进工具的最有效方法之一：对工具描述和规范进行提示工程。因为这些加载到智能体的上下文中，它们可以集体引导智能体朝向有效工具调用行为。

编写工具描述和规范时，思考你会如何向团队新成员描述你的工具。考虑你可能隐式带来的上下文——专用查询格式、小众术语定义、底层资源之间的关系——并使其显式。通过清晰描述（并用严格数据模型强制执行）预期输入和输出来避免歧义。特别是，输入参数应该明确命名：而不是名为 `user` 的参数，尝试名为 `user_id` 的参数。

通过评估，你可以更自信地衡量提示工程的影响。即使对工具描述的微小改进也能产生巨大改进。Claude Sonnet 3.5 在我们对工具描述进行精确改进后，在 SWE-bench Verified 评估上实现了最先进的性能，大幅降低错误率并提高任务完成率。

你可以在我们的开发者指南中找到工具定义的其他最佳实践。如果你为 Claude 构建工具，我们还建议阅读工具如何动态加载到 Claude 的系统提示。最后，如果你为 MCP 服务器编写工具，工具注释帮助披露哪些工具需要开放世界访问或进行破坏性更改。

## 展望未来

构建智能体的高效工具，我们需要将软件开发实践从可预测的确定性模式重新定向到非确定性模式。

通过本文中描述的迭代、评估驱动过程，我们识别了使工具成功的一致模式：高效工具是有意和清晰定义的，审慎使用智能体上下文，可以在不同工作流中组合，并使智能体能够直观解决现实世界任务。

未来，我们期望智能体与世界交互的具体机制会演变——从 MCP 协议更新到基础 LLM 升级。通过系统化、评估驱动的方法改进智能体工具，我们可以确保随着智能体变得更强大，它们使用的工具将随之演变。
