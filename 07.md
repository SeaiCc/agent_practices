# AI 智能体的有效上下文工程

发布于 2025 年 9 月 29 日

上下文是 AI 智能体关键但有限的资源。在本文中，我们探索了有效策划和管理驱动智能体的上下文的策略。

---

在提示工程成为应用 AI 关注焦点的几年之后，一个新术语开始崭露头角：**上下文工程**。使用语言模型构建正在变得不那么注重为提示找到正确的词语和短语，而更多是关于回答更广泛的问题——"什么样的上下文配置最有可能产生我们模型的期望行为？"

**上下文**指的是从大型语言模型 (LLM) 采样时包含的令牌集合。手头的**工程**问题是在 LLM 的固有约束下优化这些令牌的效用，以持续实现期望的结果。有效地驾驭 LLM 通常需要*在上下文中思考*——换句话说：考虑 LLM 在任何给定时间可用的整体状态以及该状态可能产生的潜在行为。

在本文中，我们将探索新兴的上下文工程艺术，并提供构建可操控、有效智能体的精炼心智模型。

## 上下文工程 vs. 提示工程

在 Anthropic，我们将上下文工程视为提示工程的自然演进。提示工程指的是编写和组织 LLM 指令以获得最佳结果的方法（请参阅 [我们的文档](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview) 了解概述和有用的提示工程策略）。**上下文工程**指的是在 LLM 推理期间策划和维护最佳令牌集合（信息）的策略集，包括所有可能落在提示之外的其他信息。

在使用 LLM 进行工程的早期，提示是 AI 工程工作中最大的组成部分，因为日常聊天交互之外的大多数用例都需要针对一次性分类或文本生成任务优化的提示。正如这个术语所暗示的，提示工程的主要焦点是如何编写有效的提示，特别是系统提示。然而，随着我们转向工程能够在多次推理和更长时间范围内运行的更强大智能体，我们需要管理整个上下文状态的策略（系统指令、工具、[模型上下文协议](https://modelcontextprotocol.io/docs/getting-started/intro) (MCP)、外部数据、消息历史等）。

在循环中运行的智能体会生成越来越多*可能*对下一次推理相关的数据，这些信息必须循环细化。上下文工程是从不断演变的可能的信息宇宙中策划将进入有限上下文窗口的内容的 [艺术和科学](https://x.com/karpathy/status/1937902205765607626?lang=en)。

*与编写提示的离散任务相比，上下文工程是迭代的，每次我们决定将什么传递给模型时都会发生策划阶段。*

## 为什么上下文工程对构建强大智能体很重要

尽管 LLM 速度快且能够管理越来越多的数据，但我们观察到 LLM 像人类一样，在某个点会失去专注或感到困惑。关于针锋相对的基准测试的研究揭示了 [上下文腐烂](https://research.trychroma.com/context-rot) 的概念：随着上下文窗口中令牌数量的增加，模型从该上下文中准确回忆信息的能力下降。

虽然某些模型表现出比其他模型更温和的退化，但这种特性在所有模型中都出现。因此，上下文必须被视为具有边际收益递减的有限资源。像拥有 [有限工作记忆容量](https://journals.sagepub.com/doi/abs/10.1177/0963721409359277) 的人类一样，LLM 在解析大量上下文时使用的"注意力预算"。引入的每个新令牌都会消耗一定量的这个预算，增加了精心策划 LLM 可用令牌的需求。

这种注意力稀缺源于 LLM 的架构约束。LLM 基于 [transformer 架构](https://arxiv.org/abs/1706.03762)，这使得每个令牌能够 [关注整个上下文中的每个其他令牌](https://huggingface.co/blog/Esmail-AGumaan/attention-is-all-you-need)。这导致 n 个令牌产生 n² 个成对关系。

随着上下文长度增加，模型捕获这些成对关系的能力变得紧张，在上下文大小和注意力焦点之间产生自然张力。此外，模型从训练数据分布中开发注意力模式，其中较短序列通常比长序列更常见。这意味着模型对上下文广泛依赖的经验较少，专门参数也较少。

像 [位置编码插值](https://arxiv.org/pdf/2306.15595) 这样的技术允许模型通过将其适应原始训练的较小上下文来处理更长的序列，尽管在令牌位置理解方面有所退化。这些因素创造了性能梯度而不是硬性断崖：模型在较长上下文中仍然非常强大，但与在较短上下文上的表现相比，在信息检索和长程推理方面可能显示出降低的精度。

这些现实意味着周到的上下文工程对构建强大智能体至关重要。

## 有效上下文的构成

鉴于 LLM 受限于有限的注意力预算，*好的*上下文工程意味着找到*最小的* *可能的*高信号令牌集合，以最大化某些期望结果的可能性。实施这一做法说起来容易做起来难，但在以下部分，我们概述了这一指导原则在上下文不同组件中的实践意义。

**系统提示**应该非常清晰，使用简单、直接的语言，以适合智能体的*正确层次*呈现想法。正确的层次是两个常见失败模式之间的"刚刚好"区域。在一个极端，我们看到工程师在提示中硬编码复杂、脆弱的逻辑以引出精确的智能体行为。这种方法随着时间的推移创造了脆弱性并增加了维护复杂性。在另一个极端，工程师有时提供模糊、高层次的指导，未能给 LLM 提供期望输出的具体信号或错误地假设共享上下文。最佳层次取得平衡：足够具体以有效指导行为，又足够灵活以为模型提供强大的启发式方法来指导行为。

*在光谱的一端，我们看到脆弱的 if-else 硬编码提示，在另一端我们看到过于笼统或错误假设共享上下文的提示。*

我们建议将提示组织成不同的部分（如 `<background_information>`、`<instructions>`、`## Tool guidance`、`## Output description` 等），并使用像 XML 标记或 Markdown 标题这样的技术来划分这些部分，尽管随着模型变得更有能力，提示的确切格式可能变得不那么重要。

无论您如何决定构建系统提示，您都应该努力寻求完全概述期望行为的最小信息集合。（注意最小不一定意味着简短；您仍然需要给智能体提供足够的前端信息以确保它遵循期望的行为。）最好先从测试可用最佳模型的最小提示开始，看看它在您的任务上表现如何，然后根据初始测试中发现的失败模式添加清晰的指令和示例以提高性能。

**工具**允许智能体与其环境交互并在工作中拉入新的附加上下文。因为工具定义了智能体与其信息/行动空间之间的契约，所以工具通过返回令牌高效的信息和鼓励高效的智能体行为来促进效率，这是极其重要的。

在 [为 AI 智能体编写工具——使用 AI 智能体](https://www.anthropic.com/engineering/writing-tools-for-agents) 中，我们讨论了构建 LLM 容易理解且功能重叠最小的工具。类似于设计良好的代码库的函数，工具应该是自包含的、对错误具有鲁棒性，并且在其预期用途方面非常清晰。输入参数同样应该是描述性的、无歧义的，并利用模型的固有优势。

我们看到的最常见的失败模式之一是过于臃肿的工具集，覆盖了太多功能或导致关于使用哪个工具的模糊决策点。如果人类工程师不能明确地说出在给定情况下应该使用哪个工具，就不能期望 AI 智能体做得 better。正如我们稍后将讨论的，为智能体策划最小可行工具集也可以导致在长交互中更可靠的上下文维护和修剪。

提供示例，也称为少样本提示，是我们继续强烈建议的知名最佳实践。然而，团队通常会将一堆边缘情况塞进提示中，试图阐述 LLM 应为特定任务遵循的每个可能规则。我们不推荐这样做。相反，我们建议努力策划一组多样化、典型的示例，有效地描绘智能体的期望行为。对于 LLM，示例是值千言的"图片"。

我们在上下文不同组件（系统提示、工具、示例、消息历史等）方面的总体指导是深思熟虑，保持上下文信息丰富但紧凑。现在让我们深入运行时动态检索上下文。

## 上下文检索和智能体搜索

在 [构建有效的 AI 智能体](https://www.anthropic.com/research/building-effective-agents) 中，我们强调了基于 LLM 的工作流和智能体之间的差异。自那篇文章以来，我们转向了智能体的 [简单定义](https://simonwillison.net/2025/Sep/18/agents/)：LLM 在循环中自主使用工具。

与客户合作，我们看到该领域正在汇聚到这个简单范式。随着基础模型变得更有能力，智能体的自主水平可以扩展：更聪明的模型允许智能体独立导航微妙的问题空间并从错误中恢复。

我们现在看到工程师如何思考为智能体设计上下文的转变。今天，许多 AI 原生应用采用某种形式的基于嵌入的推理前检索，为智能体呈现重要的推理上下文。随着该领域向更智能体的方法过渡，我们越来越多地看到团队用"即时"上下文策略增强这些检索系统。

而不是预先处理所有相关数据，采用"即时"方法构建的智能体维护轻量级标识符（文件路径、存储的查询、网页链接等），并使用这些引用在运行时通过工具动态加载数据到上下文中。Anthropic 的智能体编码解决方案 [Claude Code](https://www.anthropic.com/claude-code) 使用这种方法在大型数据库上执行复杂数据分析。模型可以编写针对性查询、存储结果，并利用像 head 和 tail 这样的 Bash 命令来分析大量数据，而无需将完整数据对象加载到上下文中。这种方法模拟人类认知：我们通常不记忆整个信息库，而是引入外部组织和索引系统，如文件系统、收件箱和书签，按需检索相关信息。

除了存储效率外，这些引用的元数据提供了一种有效细化行为的机制，无论是明确提供的还是直观的。对于在文件系统中运行的智能体，`tests` 文件夹中名为 `test_utils.py` 的文件的存在意味着与位于 `src/core_logic/` 中的同名文件不同的目的。文件夹层次结构、命名约定和时间戳都提供了重要的信号，帮助人类和智能体理解如何以及何时利用信息。

让智能体自主导航和检索数据也实现了渐进式披露——换句话说，允许智能体通过探索逐步发现相关上下文。每次交互都产生 informs 下一个决策的上下文：文件大小暗示复杂性；命名 convention 暗示目的；时间戳可以是相关性的代理。智能体可以逐层组装理解，仅在工作记忆中保持必要的部分，并利用笔记策略进行额外的持久性。这种自管理的上下文窗口使智能体专注于相关子集，而不是淹没在详尽但可能不相关的信息中。

当然，有一个权衡：运行时探索比检索预计算数据慢。不仅如此，还需要有见地和周到的工程，以确保 LLM 拥有有效导航其信息景观的正确工具和启发式方法。没有适当的指导，智能体可能会通过滥用工具、追逐死胡同或未能识别关键信息来浪费上下文。

在某些设置中，最有效的智能体可能会采用混合策略，预先检索一些数据以获得速度，并根据其自由裁量权进行进一步的自主探索。"正确"自主水平的决策边界取决于任务。Claude Code 是采用这种混合模型的智能体：[CLAUDE.md](http://claude.md) 文件被简单地预先放入上下文中，而像 glob 和 grep 这样的原语允许它导航环境并即时检索文件，有效地绕过陈旧索引和复杂语法树的问题。

混合策略可能更适合动态内容较少的上下文，如法律或财务工作。随着模型能力的提升，智能体设计将趋向于让智能模型智能地行动，人类策划越来越少。鉴于该领域的快速进步速度，"做最简单有效的事情"可能仍然是我们给在 Claude 之上构建智能体的团队的最好建议。

### 长周期任务的上下文工程

长周期任务要求智能体在令牌数超过 LLM 上下文窗口的动作序列中保持一致性、上下文和目标导向行为。对于跨越数十分钟到数小时连续工作的任务，如大型代码库迁移或综合研究项目，智能体需要专门技术来绕过上下文窗口大小限制。

等待更大的上下文窗口可能看起来像是一个明显的策略。但在可预见的未来，所有大小的上下文窗口都可能受到上下文污染和信息相关性问题的影响——至少在对最强智能体性能有所要求的情况下。为了使智能体能够在延长的时间范围内有效工作，我们开发了几种直接解决这些上下文污染约束的技术：压缩、结构化笔记和多智能体架构。

**压缩**

压缩是将接近上下文窗口限制的对话、总结其内容，并用摘要重新启动新上下文窗口的做法。压缩通常作为上下文工程中推动更好长期一致性的第一个杠杆。在其核心，压缩以高保真方式提炼上下文窗口的内容，使智能体能够以最小的性能下降继续。

例如，在 Claude Code 中，我们通过在消息历史传递给模型以总结和压缩最关键细节来实现这一点。模型保留架构决策、未解决的 bug 和实现细节，同时丢弃冗余的工具输出或消息。然后智能体可以使用此压缩上下文加上五个最近访问的文件继续。用户获得连续性而无需担心上下文窗口限制。

压缩的艺术在于选择保留什么与丢弃什么之间的平衡，因为过于激进的压缩可能导致微妙但关键的上下文丢失，其重要性只有后来才显现。对于实施压缩系统的工程师，我们建议在复杂的智能体轨迹上仔细调整提示。首先最大化召回率以确保压缩提示捕获轨迹中的每个相关信息片段，然后通过消除多余内容迭代以提高精度。

一个低垂果实的多余内容示例是清除工具调用和结果——一旦工具在消息历史深处被调用，智能体为什么需要再次看到原始结果？最安全的最轻量级压缩形式是工具结果清除，最近作为 [Claude Developer Platform 上的功能](https://www.anthropic.com/news/context-management) 推出。

**结构化笔记**

结构化笔记，或智能体记忆，是一种智能体定期将笔记写入上下文窗口外持久化记忆的技术。这些笔记在以后的时间被拉回上下文窗口。

这种策略以最小的开销提供持久记忆。像 Claude Code 创建待办事项列表，或您的自定义智能体维护 NOTES.md 文件，这个简单的模式允许智能体跨复杂任务跟踪进度，保持否则会在数十次工具调用中丢失的关键上下文和依赖关系。

[Claude 玩宝可梦](https://www.twitch.tv/claudeplayspokemon) 展示了记忆如何在非编码领域转变智能体能力。智能体在数千个游戏步骤中保持精确统计——跟踪目标如"在过去的 1,234 步中我一直在 1 号路线训练我的宝可梦，Pikachu 已获得 8 级，目标是 10 级"。没有任何关于记忆结构的提示，它开发探索区域的地图，记住解锁了哪些关键成就，并保持战斗策略的战略笔记，帮助它学习哪些攻击对不同对手最有效。

上下文重置后，智能体读取自己的笔记并继续数小时的训练序列或地牢探索。这种跨总结步骤的一致性实现了仅将信息保持在 LLM 上下文窗口中时不可能的长周期策略。

作为我们 [Sonnet 4.5 发布](https://www.anthropic.com/effective-context-engineering-for-ai-agents) 的一部分，我们在 Claude Developer Platform 上公开发行了 [记忆工具](http://anthropic.com/news/context-management) 公开测试版，通过基于文件的系统使在上下文窗口外存储和咨询信息更容易。这允许智能体随时间建立知识库，跨会话维护项目状态，并在不将所有内容保持在上下文中时引用以前的工作。

**子智能体架构**

子智能体架构提供了绕过上下文限制的另一条路。而不是一个智能体试图跨整个项目维护状态，专门的子智能体可以使用干净的上下文窗口处理专注任务。主智能体与高层计划协调，而子智能体执行深度技术工作或使用工具查找相关信息。每个子智能体可能会广泛探索，使用数万个或更多令牌，但仅返回其工作的浓缩、提炼摘要（通常 1,000-2,000 个令牌）。

这种方法实现了清晰的关注点分离——详细的搜索上下文保持在子智能体内孤立，而主智能体专注于综合和分析结果。这种模式在 [我们如何构建多智能体研究系统](https://www.anthropic.com/engineering/multi-agent-research-system) 中讨论，在复杂研究任务上显示出比单智能体系统实质性的改进。

这些方法之间的选择取决于任务特征。例如：

- 压缩维持需要大量来回的任务的对话流；
- 笔记记录在具有清晰里程碑的迭代开发中表现出色；
- 多智能体架构处理复杂的研究和分析，其中并行探索带来回报。

即使模型继续改进，在扩展交互中保持一致性的挑战仍将保持在构建更有效智能体的核心。

## 结论

上下文工程代表了我们要用 LLM 构建方式的根本转变。随着模型变得更有能力，挑战不仅仅是打造完美的提示——而是在每一步精心策划什么信息进入模型有限的注意力预算。无论您是为长周期任务实施压缩、设计令牌高效工具，还是使智能体能够即时探索其环境，指导原则保持不变：找到最小的高信号令牌集合，最大化期望结果的可能性。

我们概述的技术将随着模型改进而继续演进。我们已经看到更聪明的模型需要较少的规定性工程，允许智能体以更自主的方式运行。但随着能力扩展，将上下文视为宝贵的有限资源将仍然是构建可靠、有效智能体的核心。

今天就在 Claude Developer Platform 上开始上下文工程，并通过我们的 [记忆和上下文管理](https://platform.claude.com/cookbook/tool-use-memory-cookbook) 食谱访问有用的提示和最佳实践。
